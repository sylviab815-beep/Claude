# OUTPUT FORMAT

The critique is delivered in chat as Markdown, not as a separate file (unless the user explicitly asks for a file). It's a conversation move, not a deliverable.

## Template

```
[One sentence of context — pass number, genre, length target, anything inferred from the outline if not provided.]

**[Bold problem statement, located in the outline.]** [Two to four sentences explaining what's wrong, why it matters, and a concrete proposed fix.]

**[Next bold problem statement.]** [Same structure.]

[Continue for as many findings as the pass produces — typically 8-15 in pass 1, fewer in later passes.]

**Priority order for v[N+1]:**

1. [Highest-leverage fix.]
2. [Next.]
…
[Up to 15 items, ranked by how much each fix unblocks the rest of the outline.]

[Optional closing line — invite the author to flag any items they want dropped or pushed harder.]
```

## Style rules

- **Bold the problem statement first.** Not the category name. The problem itself.
- **Locate the problem.** Name the chapter, section, or beat number where the bug lives. Vague critique is useless.
- **Explain why it matters in one sentence max.** Authors discount problems they don't understand the cost of, but they don't need a paragraph.
- **Propose a concrete fix.** Specific enough that the author could fold it in without further conversation. "Compress Act I to 7 chapters by collapsing 5-9 into three" not "fix the pacing."
- **Use the user's terminology.** Mirror their cosmology names, character names, beat names. Don't translate.
- **Don't pad.** If a category is clean, skip it — don't write "this category looks fine" for every clean check.
- **Don't flatter.** Don't open with three paragraphs about what works. Open with the most important problem. The author asked for critique, not validation.
- **Don't soften with hedges.** "I might consider perhaps suggesting" reads cowardly. Be direct.
- **Don't be cruel.** Direct ≠ unkind. The tone is "tough editor who respects you," not "internet stranger who wants to feel smart."

## Length

- Pass 1: 8-15 findings, typically 1100-1500 words
- Pass 2: 6-12 findings, typically 800-1200 words (fewer items, deeper categories)
- Pass 3: 4-10 findings, typically 600-1000 words (deeper still — theme, series architecture, reader-promise)
- Pass 4+: Keep getting shorter. If pass 4 has only three new items, deliver three new items. The user values novel observations over thoroughness.

## What goes in the priority list

The priority list orders fixes by **leverage,** not severity. Some fixes unblock other fixes — those go first. Some are independent — they're ordered by impact. Some are stylistic and can wait — those go last or get omitted from the list.

Typical priority-list ranking:

1. Coherence-breaking bugs (cosmology rules that contradict, named characters who vanish)
2. Genre-fit bugs (DNF triggers, sub-genre incoherence)
3. Pacing/structure bugs that affect everything downstream
4. Character agency / arc bugs that affect specific arcs
5. Trope-checklist gaps
6. Heat-curve adjustments
7. Continuity prep (names, capitalizations)

A 15-item priority list is the upper bound. A 5-item priority list for a clean outline is appropriate.

## Iteration discipline (pass 2+)

Open the critique with one line acknowledging what changed since last pass:

> *Pass 3 — assuming v2 fixes landed. Going hard on theme, series architecture, and reader-promise.*

Then dive in. Do not rehash items the user folded in. Verify previous fixes by name only if there's evidence they didn't land.

## Sample (abbreviated, for reference)

Here's what an opening of a pass-2 critique might look like:

```
Pass 2 — RH romantasy, 90-100k target, second look on v2 with the burn-out timer in.

**The burn-out timer's window is undefined.** §1 says "roughly six weeks" but the chapter map runs Ch 20 to Ch 50 with the timer activating at Ch 37. That's 13 chapters of in-world time post-first-Tether — which at outline pace could be days or weeks. Lock the in-world calendar so the reader knows whether the clock is screaming or running. Suggested: each Tether buys roughly two weeks; first Tether starts Day 1, by Day 35 she's symptomatic, by Day 42 she's dying.

**Caspian's first-spice scene at Ch 32 has no on-page condom logic.** Not a moralizing flag — a worldbuilding-coherence one. The bond mechanic fires through skin contact during sex. If contraception interferes mechanically, it'd be a Tether-blocker the outline hasn't addressed. Either pick *Tether forms regardless* (lean tropey) or pick *bond requires barrier-free* (raises stakes for first spice in interesting ways).

[…]

**Priority order for v3:**
1. Lock the in-world calendar.
2. Decide bond-mechanic-vs-contraception rule.
3. […]
```

The opening is one line. Each finding is bold-statement → why → fix. The priority list is ranked by leverage.
